# SPEX P0 Audit and Build Optimization Plan

Status: active remediation checklist

Scope: architecture, security, documentation, tests, GitHub governance, releases, and Rust build performance

Audit snapshot: 2026-07-16

Project principle: Secure. Permissioned. Explicit.

## 1. Purpose and use of this document

This document consolidates the current SPEX audit findings and provides an ordered remediation plan. It is both a technical record and a progress checklist.

Check an item only when all of its stated completion evidence exists. A code change is not complete merely because it was implemented: its tests, documentation, security analysis, and review must also be complete.

Checklist convention:

- [ ] Not started or not demonstrated.
- [x] Completed and supported by evidence.

When checking an item, add a short note below it containing the PR, commit, test command, benchmark, ADR, or document that proves completion.

This document does not authorize protocol, wire-format, or cryptographic changes. Those changes still require an approved task, security review, tests, documentation, and human review under `AGENTS.md`.

## 2. Executive assessment

SPEX has a strong conceptual foundation:

- It separates core protocol types, group cryptography, transport, bridge, client orchestration, and CLI concerns.
- It treats transport as untrusted.
- It explicitly values canonical CBOR, signatures, permission grants, epochs, configuration hashes, and proof of work.
- It forbids unsafe code in `spex-core`.
- It includes negative tests, property tests, adversarial tests, and fuzz targets.

The implementation is not yet ready to be described as a production-grade protocol v1 implementation. The highest-risk issue is the gap between the architecture described publicly and the path used by the actual client and CLI. The main client flow uses the custom `Group` abstraction while the RFC 9420 `mls-rs` integration remains a separate path.

Current assessment:

| Area | Assessment | Main reason |
| --- | --- | --- |
| Conceptual architecture | Good | Clear crate-level separation and explicit invariants |
| Implemented architecture | Needs correction | Product path and documented MLS path diverge |
| Security controls | Needs correction | PoW, grant trust, expiry, and network-bind gaps |
| Documentation | Extensive but inconsistent | Several claims do not match current code or GitHub state |
| Tests | Good foundation, currently unhealthy | Workspace and CI do not currently pass all gates |
| CI and governance | Insufficient for a security protocol | Required checks and review policy are weaker than documented |
| Release readiness | Not ready | No release artifacts, current advisories, and broken CI |

## 3. P0 release blockers

The following items must be resolved before another production or stable release.

### 3.1 Connect the product flow to the declared MLS architecture

Evidence:

- `crates/spex-mls/src/lib.rs` contains both a custom `Group` implementation and `MlsRsClient`/`MlsRsGroup` wrappers.
- `crates/spex-client/src/lib.rs` imports and uses the custom `Group` path.
- The README describes the product as using MLS and benefiting from MLS properties.

- [ ] Decide whether RFC 9420 `mls-rs` is the normative product backend.
- [ ] Record the decision in an ADR, including migration and compatibility consequences.
- [ ] Connect the client and CLI product path to the approved backend.
- [ ] Clearly isolate, deprecate, or rename the custom group implementation if it remains for tests or migration.
- [ ] Add end-to-end tests that prove the CLI/client path uses the approved MLS implementation.
- [ ] Add interoperability and epoch-transition tests for the real product path.
- [ ] Update README and protocol documentation so they make only demonstrated claims.
- [ ] Obtain human security review before declaring MLS or forward-secrecy guarantees.

Completion evidence must identify the exact implementation used by `spex-client`, not merely prove that an MLS wrapper exists somewhere in the workspace.

### 3.2 Make the proof-of-work policy mandatory when required

Evidence:

- `accept_request_payload` validates a puzzle only when the optional field is present.
- A request producer normally includes the puzzle, but a malicious producer can construct a request without it.

- [ ] Define where the trusted `requires_pow` policy originates.
- [ ] Bind the PoW policy and minimum difficulty to a trusted invitation or local policy.
- [x] Reject a missing puzzle in the current request-acceptance path.
- [ ] Reject a puzzle below the locally required difficulty.
- [ ] Add negative tests for missing, malformed, stale, replayed, and under-difficulty puzzles.
- [ ] Document which party selects the difficulty and how downgrade is prevented.

Evidence: `crates/spex-client/src/lib.rs` now returns `ClientError::InvalidPuzzle` when a request omits PoW; `test_accept_request_payload_rejects_missing_puzzle` passed with `cargo test -p spex-client --locked` on 2026-07-17. The invitation-bound policy and the remaining negative cases are still open.

### 3.3 Define and enforce grant authority

Evidence:

- The bridge receives a `verifying_key` inside the grant payload and verifies the signature using that same key.
- This proves self-consistency but does not by itself prove that the issuer is trusted or authorized.

- [ ] Define whether a bridge grant is authorization, identity, anti-abuse evidence, or a combination of these.
- [ ] Define the issuer trust anchor or grant chain.
- [ ] Bind the grant to the expected subject, action, resource, epoch, and validity interval.
- [ ] Reject self-issued or unknown-issuer grants when the model requires an authorized issuer.
- [ ] Add adversarial tests for self-issued, substituted-key, wrong-subject, wrong-resource, and replayed grants.
- [ ] Document the trust model in the protocol and bridge deployment documentation.

### 3.4 Use expiring grants by default

Evidence:

- The normal request-acceptance flow creates grants with `expires_at: None`.
- Security documentation recommends short-lived permissions.

- [ ] Define a safe default grant lifetime.
- [ ] Require an explicit human or policy decision for a non-expiring grant.
- [ ] Enforce maximum lifetime where applicable.
- [ ] Test expiration boundaries, clock skew, renewal, revocation, and replay after expiration.
- [ ] Align CLI output and documentation with the effective expiration policy.

### 3.5 Correct bridge and P2P network binding

Evidence:

- The TLS guide describes `SPEX_BRIDGE_ADDR=127.0.0.1:3000`.
- `crates/spex-bridge/src/main.rs` currently binds to `0.0.0.0:3000`.
- The documented bridge-address variable is not used by that executable path.
- The default P2P listen address is also externally reachable despite a localhost-oriented comment.

- [x] Make the bridge bind address explicitly configurable.
- [x] Use loopback as the safe bridge default unless an approved deployment profile says otherwise.
- [x] Require an explicit configuration choice for external exposure.
- [ ] Correct the P2P comment and define the intended P2P exposure policy.
- [x] Add configuration and negative startup tests.
- [x] Update TLS and deployment documentation with the actual defaults.

Evidence: `spex-bridge` now defaults to `127.0.0.1:3000`, accepts `SPEX_BRIDGE_ADDR`, and rejects malformed values. Its binary tests passed with `cargo test -p spex-bridge --locked` on 2026-07-17.

## 4. Architecture assessment and improvement checklist

### 4.1 Architecture strengths to preserve

- [ ] Preserve `spex-core` as the dependency-light protocol foundation.
- [ ] Preserve explicit validation at every untrusted decoding boundary.
- [ ] Preserve the separation between encrypted protocol content and untrusted transport.
- [ ] Preserve deterministic serialization and hashing rules.
- [ ] Preserve negative, property-based, and fuzz testing for security-sensitive parsers.
- [ ] Preserve `#![forbid(unsafe_code)]` in security-critical crates where feasible.

### 4.2 Reduce oversized modules without creating unnecessary crates

Current high-complexity files include approximately:

- `crates/spex-client/src/lib.rs`: 2,300+ lines.
- `crates/spex-transport/src/p2p.rs`: 1,500+ lines.
- `crates/spex-bridge/src/lib.rs`: 1,300+ lines.
- `crates/spex-mls/src/lib.rs`: 1,100+ lines.

The workspace already has a useful crate structure. The first improvement should be internal modules, not additional crates.

- [ ] Split client state, requests, grants, groups, storage, and migrations into focused modules.
- [ ] Split MLS model, approved backend, extensions, and legacy/test code into focused modules.
- [ ] Split P2P behaviour, protocol, limits, replay protection, and runtime concerns.
- [ ] Split bridge routes, authorization, storage, schema, limits, and metrics.
- [ ] Keep security invariants close to the code that enforces them.
- [ ] Measure incremental-build impact before and after each structural change.

### 4.3 Separate normative canonical APIs from raw-byte helpers

Evidence:

- Public documentation makes categorical canonical-CBOR claims.
- Low-level hashing and signing helpers can still accept arbitrary byte sequences.
- A `canonical-ish` description appears in core documentation.

- [ ] Define the normative canonical encoding for every wire type.
- [ ] Introduce an explicit canonical-byte type or equivalent invariant boundary.
- [ ] Make raw-byte helpers clearly low-level and difficult to misuse.
- [ ] Add determinism and cross-version canonicalization vectors.
- [ ] Remove ambiguous terms such as `canonical-ish` from normative documentation.

### 4.4 Harden local-state persistence

Evidence:

- State encryption uses AEAD and associated data.
- Persistence writes directly to the destination path.
- Plaintext legacy state is still accepted.
- Documentation names `SPEX_STATE_PASSPHRASE`, while code uses `SPEX_STATE_PASSPHRASE_FILE`.

- [ ] Implement temp-file, flush, and atomic-rename persistence.
- [ ] Apply restrictive file permissions explicitly on supported platforms.
- [ ] Define corruption and interrupted-write recovery.
- [ ] Define rollback detection requirements for identity, grants, and epochs.
- [ ] Define and test plaintext-state migration.
- [ ] Prevent silent fallback from protected to unprotected state.
- [ ] Correct the documented passphrase mechanism and environment variable.
- [ ] Version the state envelope and password-derivation parameters.

### 4.5 Remove panic paths from untrusted boundaries

Evidence:

- `ext_cfg_hash_bytes` uses an assertion for a length constraint in a public API.

- [ ] Inventory assertions, unwraps, and expects reachable from untrusted input.
- [ ] Convert input-dependent failures to explicit typed errors.
- [ ] Add malformed-input and maximum-size tests.
- [ ] Add or extend fuzz targets for each critical public parser.

### 4.6 Version bridge database migrations

- [ ] Introduce a schema-version table.
- [ ] Make migrations ordered, transactional, and idempotent.
- [ ] Add upgrade tests from every supported schema snapshot.
- [ ] Define backup, recovery, and downgrade behavior.
- [ ] Document data-retention and cleanup policies.

### 4.7 Document the bridge capability and metadata model

Bridge reads currently rely largely on possession of identifiers or keys rather than an authenticated session.

- [ ] Document identifier entropy and enumeration resistance.
- [ ] Document which metadata the bridge can observe.
- [ ] Define retention, quota, abuse, and deletion guarantees.
- [ ] Define whether card, slot, and inbox reads are public, capability-based, or authenticated.
- [ ] Add tests for enumeration, cross-tenant access, quota bypass, and metadata leakage.

## 9. Rust build-performance analysis

### 9.1 Principles adopted from the build guide and external research

The attached build guide correctly identifies common causes of slow Rust builds: code generation, LLVM optimization, monomorphization, procedural macros, linking, cache invalidation, and large dependency trees.

The article “Fast Rust Builds” adds an important architectural principle: substantial generic implementations should not sit at crate boundaries. Ergonomic generic wrappers should be thin and should delegate immediately to concrete, non-generic implementations. This limits downstream monomorphization while keeping the public API convenient.

These recommendations are hypotheses until measured on SPEX. The plan therefore begins with baseline collection and follows the dependency graph from the most widely reused crates outward.

Primary sources:

- [Fast Rust Builds — Keeping Instantiations in Check](https://matklad.github.io/2021/09/04/fast-rust-builds.html#Keeping-Instantiations-In-Check)
- [Cargo build profiles](https://doc.rust-lang.org/cargo/reference/profiles.html)
- [Cargo features](https://doc.rust-lang.org/cargo/reference/features.html)
- [Cargo build timings](https://doc.rust-lang.org/cargo/reference/timings.html)

### 9.2 Current SPEX build characteristics

Confirmed characteristics:

- The workspace already contains six focused crates; “split into a workspace” is already implemented.
- There is no project `.cargo/config.toml` defining a linker or wrapper.
- The workspace root does not currently define custom dev or release profiles.
- CI already uses `Swatinem/rust-cache`.
- CI separately runs build, default-feature tests, and all-feature tests, creating overlapping compilation work.
- `libp2p` is a direct dependency of both `spex-transport` and `spex-cli` with a broad feature set.
- `keyring`/`keyring-core` currently introduce a large transitive subtree including database, indexing, build-script, and platform components.
- The CLI test dependency on `reqwest` enables `rustls-tls` without disabling default features, allowing native-TLS and Rustls paths to coexist.
- Multiple versions of `rand`, `rand_core`, `hashbrown`, `itertools`, `thiserror`, and other transitive crates are present.
- MLS, libp2p, keyring, SQLite bundled compilation, cryptographic crates, and procedural macros are credible first-build cost centers.

### 9.9 Phase 6 — control monomorphization at crate boundaries

Compatibility: architecturally relevant because SPEX has several public crates, but it must be profile-driven rather than speculative.

- [ ] Run `cargo llvm-lines` or an equivalent analysis on representative builds.
- [ ] Identify generic functions with many instantiations and substantial LLVM IR.
- [ ] Prioritize generic APIs exported from foundational crates and called by several downstream crates.
- [ ] Keep ergonomic generic wrappers thin.
- [ ] Delegate substantial work to concrete private functions such as `&Path`, slices, or concrete protocol types.
- [ ] Consider dynamic dispatch for cold callback paths only when runtime cost and object safety are acceptable.
- [ ] Keep generic, inlined implementations for measured hot paths where runtime performance justifies them.
- [ ] Benchmark runtime performance, binary size, and build time together.
- [ ] Avoid changing cryptographic or protocol semantics during compile-time refactors.

Illustrative pattern:

```rust
pub fn load<P: AsRef<Path>>(path: P) -> Result<State> {
    load_from_path(path.as_ref())
}

fn load_from_path(path: &Path) -> Result<State> {
    // Substantial implementation is compiled once in this crate.
}
```

The example is a design pattern, not an instruction to change a specific SPEX function without measurement.

### 9.10 Phase 7 — development and release profiles

Compatibility: applicable after baseline measurement.

- [ ] Measure the current Cargo defaults before adding custom profiles.
- [ ] Evaluate reduced debug information for local development if linking and disk use are dominant.
- [ ] Preserve sufficient debug information for security diagnostics and crash analysis.
- [ ] Keep incremental compilation enabled for normal development unless evidence supports an exception.
- [ ] Evaluate package-specific optimization only for dependencies that are unusably slow in debug mode.
- [ ] Keep release-profile performance decisions separate from development-build decisions.
- [ ] Benchmark ThinLTO only when runtime performance or binary size requires it.
- [ ] Record the reason and benchmark for every non-default profile setting.

Settings that are not build-speed recommendations:

- `lto = true` or fat LTO generally increases release build time.
- `codegen-units = 1` generally reduces parallel code generation and increases build time.
- Aggressive dependency optimization can make the first development build slower.

### 9.11 Recommendations intentionally deferred or rejected

- [ ] Do not add more crates solely because “more crates compile faster”; first measure invalidation boundaries and dependency direction.
- [ ] Do not use recurring `cargo clean` as a normal troubleshooting step.
- [ ] Do not adopt nightly-only compiler backends for required security or release gates.
- [ ] Do not use `prefer-dynamic` as a project default without deployment, portability, and runtime analysis.
- [ ] Do not enable LTO as a compile-time optimization.
- [ ] Do not set `codegen-units = 1` to make builds faster.
- [ ] Do not remove supported-platform builds or security tests to improve headline CI time.
- [ ] Do not replace audited cryptographic dependencies with local implementations for build speed.
